# divoom-control — Review (2026-06)

Covers: (1) code + architecture review (Linus / Uncle Bob lens), (2) UI/UX
review (Dieter Rams / Susan Kare lens), (3) the "rewrite the lib + daemon in
Rust?" question. Doc cleanup (4) is executed separately (see git log + `docs/`).

**Scope of confidence:** code/architecture findings are grounded in the source.
UI/UX findings are from the markup/CSS/JS structure — a *true* visual review
needs the running app; see the caveat in §2. Nothing here is hardware-verified.

A prior commit (`5de14a97`) explicitly *"modularize[d] views to ensure all files
are under 500 LOC."* It drifted back. **A rule with no guardrail is a wish.**
First recommendation, cheap and permanent:

> Add `tests/test_file_size.py` that fails if any tracked `*.py/*.c/*.js/*.css`
> under `divoom_*` exceeds 500 LOC (with an explicit, shrinking allow-list of
> known offenders). This converts "no file over 500 LOC" from aspiration into an
> enforced invariant and prevents re-drift after the splits below.

### 1.2 `gui_api.py` — the God Object (Uncle Bob: SRP, the big one)

One 921-LOC class, ~70 methods, assembled from three mixins
(`MediaSyncMixin, PresetsManagerMixin, ScannerMixin`). It owns, in one place:
window chrome (min/max/close), transport status, ~40 device-command wrappers
(lighting, clock, channels, VJ/EQ, text, alarms, sleep, timers, weather, FM,
memorial, screen, factory reset), notification routing, the menubar push, and
the daemon client. That is at least six responsibilities.

**Uncle Bob:** this is the textbook God Object. Split by responsibility into
collaborators the bridge composes (not inherits):
- `ConnectionApi` (scan/connect/status/LAN) — already mostly `ScannerMixin`.
- `LightingApi` (light/clock/channel/vj/visualization/text).
- `ToolsApi` (alarm/sleep/timer/countdown/noise/memorial/timeplan/screen/reset).
- `WidgetsApi` (weather/stocks/sysmon/cover-art) — already mostly `MediaSyncMixin`.
- `WindowApi` (chrome).
The pywebview bridge becomes a thin facade that exposes these. Each lands well
under 500 LOC and is independently testable.

**Linus:** the *data* problem behind the size. ~10 command methods repeat this
exact shape:
```python
if target_mode == "wall":
    if not self._rebuild_wall_instance(): return False
    return self._run_async(self.wall_instance.<m>(...))
target = self.current_divoom
if not target: return False
return self._run_async(target.<m>(...))
```
That branch is copy-pasted, not abstracted. Kill it with one dispatcher:
```python
def _dispatch(self, build_coro):  # build_coro(target) -> coroutine
    target = self._wall_target() or self.current_divoom
    if target is None: return False
    return self._run_async(build_coro(target))
```
Then `set_clock` is one line. ~150 LOC of duplication evaporates and the wall vs
single-device rule lives in exactly one place.

### 1.3 `daemon.py` — dispatch-by-if-ladder + four jobs in one class

- `handle_command` is a 15-branch `if command == "...":` ladder. Replace with a
  registry: `self._handlers = {"device_call": self._cmd_device_call, ...}` and
  `return self._handlers.get(cmd, self._unknown)(args)`. Adding a command stops
  meaning "edit the ladder."
- `DivoomDaemon` is simultaneously: the **device owner** (asyncio loop, connect,
  device_call, wall, scan, sync_artwork), the **notification service** (monitor,
  sink, routing), the **socket server** (Unix+TCP listeners, auth, blob
  materialization), and the **subscriber fan-out**. Four cohesive units behind
  one 730-LOC class. Extract `DeviceOwner`, `NotificationService`, and
  `SocketServer`; `DivoomDaemon` wires them. Each is then ~200 LOC and unit-
  testable in isolation (the network tests already hint at the seams).

### 1.4 Stringly-typed and tuple-typed surfaces (Linus: "show me the structs")

- `device_call` dispatches a dotted string via `getattr` chains. Fine *at the RPC
  boundary* (it's a generic proxy), but it's untyped — a typo is a runtime error,
  not a compile/lint error. Acceptable; document it as the one dynamic seam and
  don't let the pattern leak inward.
- The wall device is an unnamed 6-tuple `(divoom, x, y, size, width, height)`
  unpacked in many call sites (and several places use `entry[3]` by index).
  Make it a `@dataclass DeviceSlot`. Self-documenting, and you stop counting
  commas. This is the kind of "improve the data structure and the code follows"
  change Linus favors over cleverness.

### 1.5 Inheritance vs composition (Uncle Bob)

`DivoomGuiAPI(MediaSyncMixin, PresetsManagerMixin, ScannerMixin)` is fat-interface
inheritance where the mixins silently share state (`self.current_divoom`,
`self._client`, `self.wall_slots`). That implicit coupling is why the class is
hard to split. Prefer composition with explicit collaborators that receive a
small context object — it makes the dependencies visible and the units testable
without instantiating the whole bridge.

### 1.6 What is genuinely good (credit where due)

- The **protocol/data plane separation** is right: a pure, transport-agnostic
  framing layer; NDJSON only for control; binary kept off the socket.
- **Dual-impl native+Python with byte-for-byte correctness tests** + a conftest
  auto-rebuild guardrail. This is excellent and rare — keep it.
- The **single-owner daemon model** (R17) is the correct resolution of the
  single-BLE-owner constraint.
- The **R19 transport seam** (Unix/TCP behind one framing layer) made the network
  feature a small change — evidence the boundary was drawn well.
- Test suite: **991 passing**, with real regression coverage (no-emoji, fonts,
  pyproject, dual-impl, network auth/blobs). The discipline shows.

### 1.7 Priority order for the refactor (each = its own green, committed round)

1. `tests/test_file_size.py` guardrail (1 commit, prevents re-drift). **Do first.**
2. `gui_api` dispatcher helper (1.2 Linus fix) — removes ~150 LOC, no behavior
   change, big readability win, low risk.
3. Split `gui_api` into the API collaborators (1.2 Uncle Bob).
4. `daemon.py` command registry + extract DeviceOwner/NotificationService/
   SocketServer (1.3).
5. `DeviceSlot` dataclass (1.4).
6. Split the four >500-LOC `web_ui/*.js/css` (mechanical; tied to §2).

## 2. UI / UX review (Rams / Kare)

**Caveat first (honesty):** I reviewed the HTML/CSS/JS structure, not the rendered
product. A real Rams/Kare review judges pixels, contrast, motion, and flow on the
running app. I can do that live (launch the GUI + screenshot) if you want — this
section is principles + structure-level findings only.

### 2.1 Structure-level observations
- The UI is large: `index.html` 449 LOC of static markup + `templates.js` 718 LOC
  generating more, across Channels / Tools / Settings / Gallery / Wall / Widgets.
  `settings.js` alone is 745 LOC. That surface area is the first thing to
  interrogate against Rams.
- Styling is heavy: `style.css` 510 + `widgets.css` 524 + per-area CSS. The README
  describes a "glassmorphic" aesthetic.

### 2.2 Dieter Rams — "Weniger, aber besser" (less, but better)
- **#5 Unobtrusive / #10 As little design as possible:** glassmorphism (blur,
  translucency) is decorative by nature and fights legibility — it competes with
  the content (which, for a device controller, is controls + a live preview).
  Audit every blur/opacity layer against "does this help the user act, or just
  look modern?" Rams would strip the chrome until the controls are the loudest
  thing on screen.
- **#4 Makes a product understandable:** with six top-level areas, the question is
  whether a first-time user knows where "make my screen show album art" lives.
  Recommend a task audit: list the top ~8 user goals and count clicks/ambiguity
  to each. Consolidate areas that exist for *our* mental model, not the user's.
- **#3 Aesthetic & #6 Honest:** a controller should reflect device truth. The
  R17 cutover means the GUI now shows *daemon-reported* status — good, that's
  honest. Ensure every control reflects real device state (the get_* read-back is
  a known gap, task #20) rather than optimistic UI.

### 2.3 Susan Kare — clarity, character, craft at small sizes
- **Icon coherence:** the app mixes Divoom's imported raster EQ/VJ icons
  (`assets/icon_*.webp`) with custom tab SVGs. Kare's signature is *one coherent
  visual language*. Two icon vocabularies in one window reads as two apps.
  Recommendation: pick one system — ideally custom, legible-at-16px SVGs for app
  chrome, and treat the device's own EQ/VJ thumbnails clearly as *device content*
  (framed/labeled), not app iconography.
- **Pixel honesty:** this is a *pixel-display* controller — Kare territory. There's
  an opportunity (not a defect) to lean into a crisp, grid-true, legible visual
  identity that mirrors the device, instead of generic glass. That would make the
  product feel designed-for-purpose rather than themed.
- **The no-emoji rule + single-font-source rule (enforced by tests)** are exactly
  the kind of consistency discipline Kare embodies. Good.

### 2.4 Recommended next step for UI/UX
Let me launch the app and do a screenshot-based pass (contrast ratios, hit
targets, the six-area IA, icon coherence) — that turns §2 from principled to
specific. Flagged, not done, because it needs the running GUI.

## 3. Should the lib + daemon be rewritten in Rust?

**Short answer: rewrite the *daemon* in Rust only if you have a concrete
footprint/embedded driver; do *not* rewrite the library. For most goals, no.**

### 3.1 What Rust would actually buy
- A single static binary daemon — no Python runtime — attractive for headless
  Linux / SBC / "always-on appliance" deployments (which the R19 network server +
  R20 Linux work point toward).
- Lower idle memory + real threads; `btleplug` for cross-platform BLE.
- The native encoders (already C) become first-class Rust instead of a ctypes
  dual-impl you maintain in two languages.

### 3.2 What it would cost
- **~14.3K LOC of mature, tested protocol code** in `divoom_lib` would be
  rewritten. That code's value *is* its accumulated correctness (device quirks,
  framing edge cases) — exactly the stuff a rewrite silently breaks. High risk,
  low user-visible reward.
- The whole Python ecosystem around it goes too: the CLI, the MCP server, the
  pywebview bridge, the test suite (991 tests). The GUI *cannot* move (pywebview
  + JS), so you'd permanently straddle two languages with an FFI/IPC seam.
- `bleak` → `btleplug` is not free: different backends, different timing/quirk
  behavior on exactly the devices you've already tuned for.

### 3.3 The nuance that matters
The two components have opposite profiles:
- **`divoom_daemon` is small (1,845 LOC), long-running, and the place where
  footprint/perf/uptime actually matter — and it already has a clean socket
  protocol boundary.** That is a *good, low-risk* Rust candidate: you could
  reimplement it behind the identical NDJSON/TCP protocol and the GUI wouldn't
  know.
- **`divoom_lib` is large, mature, tested, and perf-insensitive except for the
  hot paths that are already native C.** Rewriting it is the worst risk/reward in
  the codebase.

### 3.4 Recommendation
1. **Default: stay on Python.** "Lightweight" is better achieved by trimming
   optional deps and packaging (PyInstaller/`uv`/a slim venv) than by a rewrite.
   The CPU-bound work is already in C.
2. **If (and only if) you have an embedded/appliance target:** rewrite *only the
   daemon* in Rust against the existing protocol, expose the existing C as a Rust
   crate (`cc`/`bindgen`), and keep `divoom_lib`/CLI/GUI in Python for desktop.
   The R19 protocol boundary is what makes this feasible without a big bang.
3. **Do not** rewrite the library wholesale. The ROI isn't there and the
   regression risk is high.

A cheap way to de-risk a future decision: a small Rust daemon *spike* that speaks
the protocol for `ping`/`device_status` only, measured against the Python daemon's
footprint. Decide with numbers, not vibes.
